# Cline Rules（统一规则集）

版本：v0.1（草案）

目的
----
统一项目级别的业务开发规范，降低耦合、避免硬编码、提高可维护性与审查效率。

核心原则（MUST / SHOULD）
-------------------------
- 低耦合 — MUST：模块间通过接口/服务/事件交互，禁止依赖实现细节。  
- 单一职责 — MUST：类/方法应只负责一件事，便于理解与测试。  
- 避免魔法数字 — MUST：业务常量集中管理（constants/ 或配置），使用有意义的命名或枚举。  
- 错误可见性 — MUST：异步操作须显式处理错误，并向用户/监控暴露可理解信息。  
- 简洁与可维护 — SHOULD：优先简单可读实现，避免过度抽象与深度嵌套。

具体规则（格式：规则 — 等级 — 说明 — 检查点）
--------------------------------------------------
1. 低耦合 — MUST  
   说明：模块仅通过接口/事件/service 交互，内部实现不对外暴露。  
   检查点：是否存在跨模块直接访问实现类或内部状态。

2. 业务常量集中管理 — MUST  
   说明：所有状态值、类型、超时、API 地址等不得裸写在业务代码中。  
   检查点：常量是否集中在 constants/ 或配置文件并使用枚举或常量对象。

3. 单一职责与函数限制 — MUST / SHOULD  
   说明：函数应只做一件事，建议不超过 50 行，参数不超过 5 个；复杂逻辑拆分为服务/工具。  
   检查点：是否存在超长或多职责函数。

4. 命名与可读性 — MUST  
   说明：变量、方法、类名需表达业务意图（如 fetchUser、submitOrder）。  
   检查点：是否存在无意义或混淆命名（arr1、tmp 等）。

5. 控制副作用 — MUST  
   说明：副作用应显式、可控，UI 组件应尽量避免内嵌大量业务逻辑。  
   检查点：组件内是否包含未经封装的业务实现。

6. 抽象与复用策略 — SHOULD  
   说明：以“三次重复”作为提取复用的参考阈值；避免为未知需求过早抽象。  
   检查点：是否为少量重复创建复杂抽象。

7. 错误处理与用户提示 — MUST  
   说明：异步操作需捕获错误并提供友好用户提示与必要的监控埋点。  
   检查点：异步调用是否有 try/catch 或统一错误处理层。

8. 代码审查与例外流程 — SHOULD  
   说明：规则按等级区分，例外需在 PR 中声明理由并指明审批人。  
   检查点：PR 是否包含例外说明与审批记录（如适用）。

可执行化与落地建议
--------------------
- 将 MUST 条目映射为可自动校验项（建议采用 ESLint/TypeScript 规则、静态分析脚本或 CI 检查）。  
- 每条规则附带 Code Review 检查点清单，便于审查时验证。  
- 提供 PR 模板，强制在需要例外时填写理由与审批人。  
- 文档需包含版本与 CHANGELOG，所有变更须在 PR 中说明影响范围。

维护与变更流程
----------------
- 文档采用语义化版本，修改时更新版本号与 CHANGELOG。  
- 规则变更需经架构/维护团队审批；重大变更需评估影响并通知相关团队。  
- 例外由提交 PR 的作者在 PR 描述中说明，审批人在合并前给出批准记录。
