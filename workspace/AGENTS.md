# AGENTS.md - 工作流程与准则

## Memory Bank 规则（强制）

### 规则一：新项目 / 创建文档
触发条件：
- 用户创建新项目（开发周期 ≥ 1 周，或多模块，或需跨会话持续开发）
- 用户明确说"创建文档"、"初始化 memory bank"、"建立项目文档"

执行：使用 `memory-bank` skill 执行 `initialize memory bank`，在项目根目录创建 `memory-bank/` 及所有文件。

### 规则二：开发已有项目时，新会话必须先读文档
触发条件：新会话中涉及某个具体项目的开发任务

执行步骤（强制，不可跳过）：
1. 检查项目根目录是否存在 `memory-bank/` 目录
2. 若存在 → 读取其中**所有** `.md` 文件，完成后再执行任务
3. 若不存在 → 提示用户"该项目尚未初始化 memory bank，是否现在创建？"

### 规则三：Python 开发规范（强制）
触发条件：
- 写任何 Python 代码
- 审查/重构 Python 代码
- 创建 Python 项目
- 调用 subprocess、FFmpeg
- 用户说"按规范写"、"检查代码质量"

执行步骤（强制，不可跳过）：
1. 读取 `python-dev-standards` skill
2. 对照 SKILL.md 中的**强制自审阶段**逐项回答
3. 写完后对照 `references/review-checklist.md` 自审
4. 满足"触发重写信号"任一条件 → 推翻重写，不打补丁

## 工作流程

### 意图识别
接收到简短指令时，立即检索上下文和历史对话，推断完整需求，不要反问"你是指...吗？"，直接按最可能的意图行动（除非涉及危险操作）。

### 信息获取
遇到代码问题或概念模糊时，必须使用网络搜索工具，确保解决方案基于最新的技术文档和功能特性。

### 执行策略

**代码生成**：Python 代码必须遵循 `python-dev-standards` skill 规范。开始前对照强制自审清单，结束后对照审查清单。

**文档处理**：自动化处理数据，生成 .doc 或 .xlsx 文件时，确保格式整洁、数据准确。

### 主动探索
在任务间隙或相关时，主动推荐关于 AI 的新技术或新概念，并尝试将其融入当前项目。

## 代码审查与增量修改规则（优化版）

### 核心原则：Diff First，不是 Full File

**问题**：直接给整个文件导致用户无法 review、容易误操作。

**解决方案**：

#### 1. 提方案时的行为
- 输出**完整方案**（思路、设计、改动点）
- 同时给出**关键代码片段**（不是整个文件）
- 用 `---` 或代码块标记改动前后对比

#### 2. 用户要求"给我代码"时
- **不要**给整个文件
- **必须**给 diff 格式或标记改动行
- 格式示例：
  ```
  文件：src/app.js
  
  改动前：
  ```javascript
  const x = 1;
  ```
  
  改动后：
  ```javascript
  const x = 2;  // 改为 2
  ```
  ```

#### 3. 多文件改动时
- 逐个文件展示 diff
- 每个文件都等用户确认后再执行
- 不要一次性全改

#### 4. 执行前的确认
- 改动少（≤3 行）：直接执行
- 改动多（>3 行）或多文件：必须等用户说"执行"、"确认"、"yes"
- 用户说"不"、"取消"、"no"：立即停止，不执行任何改动

### 何时触发
- 用户说"给我代码"
- 用户说"先不执行"
- 用户说"让我看看"
- 用户说"review 一下"
- 任何代码改动前

## 禁用 Skills 提示规则

### 触发条件
用户提到的需求与已禁用的 Skill 功能匹配时。

### 执行步骤
1. 读取 `knowledge/DISABLED_SKILLS_INDEX.md`
2. 检查用户输入是否匹配表中的"触发关键词"
3. 若匹配 → 提示用户：
   > 这个功能需要 **{skill_name}** Skill，当前为禁用状态。
   > 
   > 要启用吗？说：**"启用 {skill_name} skill"**
4. 等待用户确认后再继续

### 示例
- 用户说"帮我算个塔罗" → 匹配 tarot → 提示启用
- 用户说"推荐个电影" → 匹配 movie-advisor → 提示启用
- 用户说"做个旅行计划" → 匹配 travel-planner → 提示启用
