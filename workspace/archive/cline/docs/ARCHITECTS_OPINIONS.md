# 三个架构师对 Cline Rules 的具体意见

## 🤖 架构师 1：前端架构师的具体意见

### Vue 3 组件规则

**规则 1.1：Props 必须定义类型和默认值**
- ✅ 正确，保留

**规则 1.2：Emits 必须定义事件类型**
- ✅ 正确，保留

**规则 1.3：使用 `<script setup>`**
- ✅ 正确，保留

**规则 1.4：v-for 必须有 `:key`**
- ✅ 正确，保留

**规则 1.5：样式必须使用 `scoped`**
- ❌ 问题：scoped 有性能开销，且无法覆盖子组件
- 💡 意见：应该允许 CSS Modules 作为替代方案
- 💡 意见：全局样式（reset、主题）需要例外

**规则 1.6：组件必须有 `name` 属性**
- ⚠️ 问题：`<script setup>` 中 name 不是必须的
- 💡 意见：只在需要时添加（keep-alive、递归、调试）
- 💡 意见：可以用 Vite 插件自动推断

### TypeScript 类型规则

**规则 2.1：禁止使用 `any`**
- ⚠️ 问题：过于绝对，某些场景确实需要
- 💡 意见：应该改为"严格限制"，建立白名单机制
- 💡 意见：允许显式 `type TODO = any` 或 `type ThirdParty = any`

**规则 2.2：函数参数必须有类型注解**
- ✅ 正确，保留

**规则 2.3：函数返回值必须有类型注解**
- ⚠️ 问题：TS 可自动推断，强制注解可能冗余
- 💡 意见：公共 API 强制，内部函数允许推断

**规则 2.4：优先使用接口定义对象类型**
- ✅ 正确，保留

**规则 2.5：使用 `as const` 定义常量对象**
- ✅ 正确，保留

### 错误处理规则

**规则 3.1：异步代码必须使用 try-catch**
- ⚠️ 问题：粒度问题，容易导致代码冗余
- 💡 意见：应该分层处理
  - API 层：统一拦截（axios interceptors）
  - 业务层：只处理需要特殊处理的错误
  - UI 层：统一错误边界

**规则 3.2：必须有错误状态**
- ✅ 正确，保留

**规则 3.3：必须有加载状态**
- ✅ 正确，保留

**规则 3.4：必须有用户可见的错误提示**
- ✅ 正确，保留

### 避免硬编码规则

**规则 4.1：API 地址不能硬编码**
- ✅ 正确，保留

**规则 4.2：API 端点必须使用常量**
- ✅ 正确，保留

**规则 4.3：错误消息必须使用常量**
- ✅ 正确，保留

**规则 4.4：魔法数字必须使用常量**
- ⚠️ 问题：需要区分语义化数字和纯数字
- 💡 意见：需要常量的（有业务语义）：MAX_RETRY_COUNT、DEBOUNCE_DELAY_MS、ITEMS_PER_PAGE
- 💡 意见：不需要常量的（自解释的）：`value * 2`、`arr.length - 1`

### DRY 原则规则

**规则 5.1：相同逻辑出现 2 次必须提取为函数**
- ⚠️ 问题：2 次太早，容易过度抽象
- 💡 意见：改为 3 次规则更合理

**规则 5.2：API 调用模式重复必须提取为 Composable**
- ✅ 正确，保留

**规则 5.3：相似的组件必须参数化为一个组件**
- ⚠️ 问题：过度参数化导致组件臃肿
- 💡 意见：优先使用组合（slots + 原子组件），而非条件渲染（props 控制分支）

### 项目结构规则

- ✅ 正确，保留

### 缺失的规则

**缺失 1：状态管理规则**
- 💡 意见：应该明确状态分层
  - 服务端状态 → TanStack Query / SWR
  - 客户端全局状态 → Pinia（跨页面共享）
  - 组件本地状态 → ref/reactive（不提升）
  - URL 状态 → 路由参数（可分享、可刷新）

**缺失 2：性能规则**
- 💡 意见：应该包含
  - 大数据列表 → 虚拟滚动
  - 复杂计算 → useMemo（或 computed + 缓存策略）
  - 事件处理 → 防抖/节流
  - 组件渲染 → v-once / v-memo 明确使用
  - 图片/资源 → 懒加载 + 响应式图片

**缺失 3：测试规则**
- 💡 意见：应该包含
  - 工具函数 → 单元测试（覆盖率 > 80%）
  - Composables → 组合式测试（@vue/test-utils）
  - 组件 → 快照测试 + 交互测试
  - 关键用户路径 → E2E 测试（Playwright）

**缺失 4：导入/模块规则**
- 💡 意见：应该包含
  - 使用路径别名（@/）
  - 目录统一导出（index.ts）
  - 避免循环导入

**缺失 5：代码审查清单**
- 💡 意见：应该包含 PR 审查的具体检查项

## 🤖 架构师 2：代码质量专家的具体意见

### 规则质量评分

**高质量规则（⭐⭐⭐⭐⭐）**
- Props 类型 + 默认值
- v-for 必须有 :key
- 禁止使用 any
- 异步代码必须使用 try-catch

**中等质量规则（⭐⭐⭐⭐）**
- 必须有错误状态
- 必须有加载状态
- DRY 原则

**需要改进的规则（⭐⭐⭐）**
- 函数返回值必须有类型注解（冗余）
- 组件必须有 name 属性（不必须）
- 样式必须使用 scoped（全局样式有时必要）

### 可执行性分析

**高度可自动化（ESLint/Prettier）**
- 禁止 any → @typescript-eslint/no-explicit-any
- v-for 必须有 :key → vue/require-v-for-key
- Props 类型定义 → vue/require-default-prop
- 函数参数类型 → @typescript-eslint/explicit-function-return-type

**需要人工审查**
- 错误状态是否有用户提示
- DRY 原则判断
- "相似组件"的判断
- API 调用模式重复

### 规则冲突

**冲突 1：类型注解 vs 类型推断**
- 💡 意见：强制返回值注解与 TS 最佳实践"信任推断"冲突
- 💡 意见：应该区分公共 API（强制注解）和内部函数（允许推断）

**冲突 2：组件 name vs script setup**
- 💡 意见：`<script setup>` 本身不支持直接定义 name，需额外配置
- 💡 意见：应该改用 `defineOptions({ name: 'Xxx' })`

### 规则重复

**重复 1：API 地址 + API 端点**
- 💡 意见：两条规则本质相同，应该合并为"API 相关配置必须集中管理"

**重复 2：错误消息常量 + 错误状态**
- 💡 意见：可合并为"错误处理统一规范"

### 缺失的关键规则

**缺失 1：测试相关（严重缺失）**
- 💡 意见：应该包含
  - 单元测试覆盖率要求：工具函数 100%、Composables > 80%
  - 测试命名规范：describe/it 必须描述清楚测试场景

**缺失 2：性能相关**
- 💡 意见：应该包含
  - 大型列表必须使用虚拟滚动（列表项 > 100）
  - 避免在模板中使用复杂计算

**缺失 3：安全相关**
- 💡 意见：应该包含
  - 用户输入必须清理（使用 DOMPurify）
  - 敏感信息禁止前端暴露

**缺失 4：可访问性（A11y）**
- 💡 意见：应该包含
  - 图片必须有 alt 属性
  - 交互元素必须有键盘支持
  - 颜色对比度必须符合 WCAG AA 标准
  - 表单必须有 label 关联

**缺失 5：代码复杂度控制**
- 💡 意见：应该包含
  - 函数复杂度限制：圈复杂度 < 10
  - 函数长度 < 50 行
  - 文件大小 < 300 行
  - 组件 props < 10 个

**缺失 6：Git 提交规范**
- 💡 意见：应该包含
  - 提交信息格式：feat/fix/docs/style/refactor/test/chore
  - 禁止直接提交到 main

**缺失 7：文档要求**
- 💡 意见：应该包含
  - 公共函数必须有 JSDoc
  - 组件必须有使用示例

### 衡量指标

**代码质量指标**
- 💡 意见：Bug 密度应该 < 0.5 bugs/KLOC
- 💡 意见：Code Review 发现的问题数应该趋势下降
- 💡 意见：技术债务工时应该 < 15%

**类型安全指标**
- 💡 意见：any 使用次数应该 = 0
- 💡 意见：类型覆盖率应该 > 95%
- 💡 意见：隐式 any 数量应该 = 0

**测试指标**
- 💡 意见：测试覆盖率应该 > 80%
- 💡 意见：变异测试得分应该 > 70%
- 💡 意见：不稳定测试比例应该 = 0

**维护性指标**
- 💡 意见：平均圈复杂度应该 < 10
- 💡 意见：重复代码比例应该 < 5%

**规则遵守度**
- 💡 意见：ESLint 错误数应该 = 0
- 💡 意见：ESLint 警告数应该趋势下降
- 💡 意见：提交前检查失败率应该 < 5%

## 📊 三个架构师的共同意见

### 必须保留的规则
- Props 类型 + 默认值
- Emits 定义
- `<script setup>`
- v-for key
- 禁止 any
- API 配置
- DRY 原则

### 必须改进的规则
- 样式 scoped → 允许 CSS Modules
- 组件 name → 改为推荐
- 返回值类型 → 区分公共 API 和内部函数
- DRY 原则 → 改为 3 次规则
- 魔法数字 → 区分语义化和纯数字
- 相似组件 → 优先组合优于继承
- 错误处理 → 分层处理

### 必须补充的规则
- 测试规范（严重缺失）
- 性能规范
- 安全规范
- 可访问性规范
- 代码复杂度控制
- Git 提交规范
- 文档规范
- 代码审查规范

### 需要量化的指标
- Bug 密度 < 0.5 bugs/KLOC
- 类型覆盖率 > 95%
- 测试覆盖率 > 80%
- 重复代码 < 5%
- 圈复杂度 < 10
- ESLint 错误 = 0
