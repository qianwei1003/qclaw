# Cline 核心开发守则

> 模块边界、命名、常量管理与职责划分

## 核心原则

| 原则 | 级别 | 说明 |
|------|------|------|
| 低耦合 | MUST | 模块间通过接口/服务/事件交互，避免直接依赖实现 |
| 单一职责 | MUST | 类/函数只负责单一业务职责 |
| 避免魔法数字 | MUST | 业务常量集中管理，禁止裸写数字/字符串 |
| 清晰可维护 | SHOULD | 命名清晰，复杂处添加注释 |

## 具体规则

### R001 - 低耦合
- 通过服务契约或接口调用，而非直接 new/导入实现类
- 遵循仓库/模块分层与域边界

### R002 - 业务常量集中管理
- 状态码、超时、API 地址等应集中管理
- 禁止在业务代码中硬编码

### R003 - 单一职责与方法长度
- 函数聚焦单一动作
- 建议：max 50 行 / max 5 参数

### R004 - 命名规范
- 避免 `arr1` / `tmp` / `doStuff` 等无意义命名
- 遵循项目命名约定（camelCase / PascalCase）

### R005 - 控制副作用
- 副作用（网络、DB、全局状态）应集中封装
- UI 层仅负责渲染

### R006 - 抽象与复用策略
- 至少 3 处重复后再提取复用
- 避免为少量重复创建复杂抽象

### R007 - 最小改动原则 ⭐
- **只修改用户明确要求的代码范围**
- 禁止顺手重命名、调整顺序、删除注释、修复 lint

### R008 - 改动报告机制
- 每次改动必须生成报告
- 包含：文件/行号、改动类型、原因
- 列出被检测但未执行的建议项

## 改动报告示例

**正确**：
```
UserList.vue:45-50 新增搜索（用户要求）
UserList.vue:23 建议 v-show（已记录但未改动）
```

**错误**：
```
未在 PR 中说明改动范围或未提交改动报告
```
