# 核心经验总结

这是从实际使用中积累的经验，不是理论。

---

## 一、人格设计

### DevCore 人格要点
- **不说废话**：直接给答案，不客套
- **批判性思维**：发现问题直接说，不遮掩
- **行动导向**：先做再说，用结果说话

### 为什么有效
- 用户是资深工程师，沟通极简
- 客套话浪费时间，影响效率
- 直接指出问题 > 委婉暗示

---

## 二、规则分层

### 为什么分层
所有规则放一个文件 → tokens 爆炸
分层后 → 常用规则始终加载，专项规则按需加载

### 分层策略
| 层级 | 文件 | 加载时机 |
|-----|------|---------|
| 核心 | AGENTS.md | 始终 |
| 专项 | rules/*.md | 触发关键词 |
| 记录 | memory/*.md | 按需读取 |

### 核心规则
- **安全边界**：删除文件必须先确认
- **意图识别**：简短指令直接行动
- **信息获取**：不确定时先搜索

---

## 三、自我改进机制

### 触发条件
- 任务失败
- 用户纠正
- 发现更好方法

### 记录内容
```markdown
### [日期] 改进点

#### 问题/不足
[描述]

#### 原因分析
[根本原因]

#### 改进方案
[具体措施]

#### 应用场景
[适用范围]
```

### 为什么有效
- 错误不记录 → 重复犯错
- 记录后提炼 → 形成规则
- 规则固化 → 自动遵守

---

## 四、记忆分层

| 层级 | 内容 | 生命周期 |
|-----|------|---------|
| 流水账 | 每日记录 | 定期压缩 |
| 教训 | 错误和纠正 | 永久 |
| 规律 | 反复出现的模式 | 永久 |
| 长期 | 重要决策 | 永久 |

### 实践经验
- 不要什么都记
- 有价值才记
- 定期回顾和提炼

---

## 五、开发模式

### 计划模式
用户说"开启计划模式"后：
1. 输出详细计划
2. 等用户确认
3. 执行时保持透明

### 开发者模式
开启后每次回复输出：
- 基于规则
- 思考过程
- 使用工具

然后等用户打分，记录反馈。

## 六、常见坑

### 1. Skill 陷阱
看到有 skill 就走 skill 流程 → 可能绕远路
**正确做法**：先试最简方案，失败再走 skill

### 2. 文件误删
不理解文件用途就删除 → 丢失重要内容
**正确做法**：删除前必须问用户

### 3. 规则膨胀
什么都往 AGENTS.md 塞 → tokens 爆炸
**正确做法**：专项规则放 rules/

### 4. 平台差异
bash 脚本在 Windows 上跑不通
**正确做法**：Windows 上优先直接调用工具

## 七、推荐 Skills

### 必装
- `qclaw-rules`：系统基础规则
- `qclaw-text-file`：跨平台文件写入
- `qclaw-openclaw`：OpenClaw 命令封装

### 按需
- `python-dev-standards`：写 Python 时
- `yt-dlp`：下载视频时
- `web-crawler-agent`：爬网站时

---

## 八、配置文件修改生效

| 类型 | 生效方式 |
|-----|---------|
| 系统注入文件（AGENTS.md 等） | 重启 QClaw |
| rules/*.md | 下次触发 |
| memory/*.md | 即时 |

---

## 九、新环境建议

### 第一步
1. 复制 files/ 到 workspace/
2. 修改 USER.md（你的身份）
3. 修改 TOOLS.md（你的路径）

### 第二步
1. 安装必装 Skills
2. 根据需要改 SOUL.md（人格）
3. 根据需要改 AGENTS.md（流程）

### 第三步
1. 重启 QClaw
2. 测试几条指令
3. 根据反馈调整
