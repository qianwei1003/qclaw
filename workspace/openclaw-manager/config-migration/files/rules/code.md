# 代码类规则

## 代码审查

### 触发条件
- 用户说"审查代码"、"review 一下"
- 写完代码后自审

### 审查流程
1. 审查代码，发现问题直接说
2. 发现基础逻辑错误或设计模式过时时 → 告知"建议重写"
3. 自我修正：发现更好实现方式 → 主动替换

---

## Python 开发规范

### 触发条件
- 写任何 Python 代码
- 审查/重构 Python 代码
- 创建 Python 项目
- 调用 subprocess、FFmpeg
- 用户说"按规范写"、"检查代码质量"

### 执行步骤
1. 读取 `python-dev-standards` skill
2. 对照 SKILL.md 中的**强制自审阶段**逐项回答
3. 写完后对照 `references/review-checklist.md` 自审
4. 满足"触发重写信号"任一条件 → 推翻重写，不打补丁

## Diff First 规则

### 核心原则
直接给整个文件导致用户无法 review、容易误操作。

### 触发条件
- 用户说"给我代码"
- 用户说"先不执行"
- 用户说"让我看看"
- 任何代码改动前

### 执行方式
1. 提方案时 → 输出完整方案 + 关键代码片段（不是整个文件）
2. 多文件改动 → 逐个展示 diff，每个等确认后再执行
3. 改动少（≤3 行）→ 直接执行
4. 改动多（>3 行）或多文件 → 必须等用户说"执行"、"确认"、"yes"才动手

### 格式示例
```
文件：src/app.js

改动前：
```javascript
const x = 1;
```

改动后：
```javascript
const x = 2;  // 改为 2
```
```
