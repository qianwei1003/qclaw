### 2026-03-28 视频下载绕远路 — 过度依赖 skill 框架

#### 问题/不足
用户要求下载 B 站视频 BV1eE411g7T2，yt-dlp 和 ffmpeg 已安装，一个命令就能搞定。我却：
1. 读了 yt-dlp skill 的 SKILL.md → 发现 bash 脚本 → 在 Windows 上硬跑 → 失败
2. 读 bash 脚本源码 → `where yt-dlp` 返回空 → 误判"没装"
3. 用 pip 装 yt-dlp → already satisfied（本来就有）
4. 装新 skill（video-downloader-skill）→ 读它的 SKILL.md
5. 最后直接跑 `yt-dlp <url>` → 瞬间成功

一个命令的事，搞了 6 步，耗时 6 分钟。

#### 原因分析
1. **Skill 优先陷阱**：看到 skill 存在就先走 skill 流程，没判断是否适用于当前平台
2. **bash 脚本不跨平台**：skill 的 download.sh 是 Linux 写的，Windows 不适用
3. **`where` 找不到 ≠ 没装**：pip user install 的 yt-dlp Scripts 目录可能不在当前 PATH，但 Python 可以直接调用。应该直接试 `yt-dlp --version` 而不是信任 `where`
4. **沉没成本驱动**：每一步失败后不是退回重新评估，而是继续"修补"，越陷越深
5. **忽视 TOOLS.md**：工具已在系统 PATH 中可用，不需要 skill 脚本包装

#### 改进方案
1. **先试最简方案**：收到下载任务 → 直接 `yt-dlp <url>` → 失败再排查
2. **Skill 脚本不是唯一路径**：skill 提供参考，但 Windows 上优先直接调用工具本身
3. **`where` 找不到时试运行**：不要只靠 `where`/`which` 判断，直接跑一下更可靠
4. **5 步自检**：执行过程中如果超过 3 步还在"准备"，停下来问自己"有没有更直接的方式"

#### 应用场景
- 所有"用 XX 工具做 YY"的任务
- Windows 环境下的 bash 脚本 skill
- 工具链已确认存在时的任务执行

### 2026-03-28 误删 memory-bank 项目文档

#### 问题/不足
删除了 ai-video-editor 项目的 memory-bank 文档系统（共10个文件），理由是"文件太多太乱"

#### 原因分析
1. **不理解 memory-bank 的用途** - 它是专门给项目用的开发文档系统，不是杂乱文件
2. **误判文件价值** - 用"整洁"代替"有价值"，没有仔细考虑每个文件的用途
3. **没有先问用户** - 直接执行删除操作，而不是先征求用户意见
4. **依赖"可以恢复"** - 以为 git checkout 可以恢复，但本身就是浪费时间

#### 改进方案
1. **删除文件前必须先问用户** - "我建议删除这些文件，因为...，你同意吗？"
2. **不轻易判断"杂乱"** - 先理解文件的用途，再决定是否删除
3. **区分"项目文档"和"临时文件"** - memory-bank 是项目的一部分，不能随便删除
4. **养成备份习惯** - 删除前先确认是否有备份价值

#### 应用场景
- 删除任何文件之前
- 判断文件是否"杂乱"之前
- 处理项目文档之前

### 2026-03-27 工作区文件按需加载改造

#### 问题/不足
- 6 个必须加载的文件（AGENTS.md、SOUL.md、USER.md、IDENTITY.md、TOOLS.md、HEARTBEAT.md）总大小 ~10k 字符
- 每次新窗口都一次性注入，导致 ~20k+ tokens 固定开销
- 规则内容重叠：SOUL.md 和 USER.md 都定义了偏好，AGENTS.md 包含大量不常用的流程规则

#### 原因分析
- 平台（OpenClaw）只在会话创建时注入文件，没有"关键词触发动态加载"的机制
- 所有规则都放在必须加载的文件里，不区分"始终需要"和"按需"
- Skills 列表全量注入（平台层面，无法绕过）

#### 改进方案
1. 精简 6 个必须文件的内容：
   - SOUL.md：~2157 → 901 字符（-58%）
   - AGENTS.md：~6655 → 1403 字符（-79%）
   - HEARTBEAT.md：~514 → 190 字符（-63%）
   - USER.md、IDENTITY.md、TOOLS.md 保持不变

2. 创建 rules/ 目录，按需加载：
   - rules/code.md - 代码审查 + Python 规范 + Diff 规则
   - rules/plan.md - 计划模式流程
   - rules/memory.md - Memory Bank 规则

3. AGENTS.md 改为"规则索引"：
   - 保留安全边界
   - 保留通用能力
   - 规则索引指向 rules/ 目录的具体文件

#### 应用场景
- 新会话创建时：固定开销从 ~10k 字符降至 ~4k 字符
- 代码类任务：额外加载 rules/code.md（+1.5k）
- 计划模式：额外加载 rules/plan.md（+0.8k）
- Memory Bank：额外加载 rules/memory.md（+1.2k）

### 2026-03-25 Python 代码质量规范缺失

#### 问题/不足
开发了完整的 AI 视频剪辑工具，但代码存在多处硬伤：
- `_build_command` 的 `-y` 放错位置
- subprocess 错误信息提取用 `[-3]` 行数猜测
- `get_video_duration` 正则 `\d{2}` 导致 >59分钟视频崩溃
- 三个模块各写各的 duration 解析，没有复用

#### 原因分析
1. 现有 `.clinerules` 规范是 TypeScript/React 导向，对 Python 项目不适用
2. 没有在写代码前触发规范检查
3. 没有 FFmpeg subprocess 调用的专门规范
4. 流程规范（memory bank、diff 修改）和代码质量规范是脱节的

#### 改进方案
1. 创建 `python-dev-standards` skill，包含：
   - FFmpeg subprocess 调用规范
   - Python 惯用写法（PEP8）
   - 错误设计规范
   - 代码审查清单
   - subprocess 封装模板
2. 在 AGENTS.md 中添加强制规则：写 Python 代码前必须读取 skill
3. 触发"重写"而非"修补"的明确信号

#### 应用场景
- 所有 Python 代码开发
- FFmpeg subprocess 调用
- 调用 subprocess 的任何工具

### 2026-03-22 学习 ClawHub Skills 安装

#### 问题/不足

- ClawHub Skills 安装
- 任何有 API 限流的操作
