# Cline 配置方案 - 根据用户反馈调整

## 📋 用户需求总结

### 核心场景
1. **新项目** - 按最流行的架构搭建规范
2. **老项目新功能** - 如何在现有代码基础上保持规范

### 优先级排序

#### 🔴 必须做（精细、完善）
- **可复用性检查** - 避免重复代码
- **硬编码检查** - 使用配置文件
- **函数设计规范** - 单一职责、参数清晰
- **组件设计规范** - Props、Emits、职责清晰
- **重复代码识别** - 提取可复用逻辑

#### 🟡 应该做（根据项目调整）
- **系统提示词** - 可以加，具体内容再讨论
- **代码规范检查** - 根据当前项目的 `.eslintrc` 调整
- **项目结构规范** - 新项目用最流行架构，老项目保持现状

#### 🟢 可以后续加
- **MCP 工作流** - 后面再加
- **Hooks** - 后面再加
- **Skills** - 如果有非常实用的先加
- **Git 相关** - 先不管

### 关键要求
- ✅ 使用第三方插件前，必须先下载当前版本文档
- ✅ 参考线上文档、node_modules 或用户编写的文档
- ✅ 避免写错配置

- 有一个新项目可以拿来测试工作流

## 🎯 新的执行计划

### 第一步：了解新项目信息

需要你提供：
1. **项目名称和类型** - 什么项目？
2. **技术栈** - Vue 2/3？TypeScript？
3. **当前的 .eslintrc 配置** - 已有的规范是什么？
4. **项目结构** - 现在的目录结构是怎样的？
5. **主要痛点** - 目前代码中最常见的问题是什么？

### 第二步：创建针对性的配置

基于新项目的实际情况，创建：

1. **系统提示词** - 针对你的项目定制
   - 新项目：按最流行架构
   - 老项目：保持现有风格，新功能遵循规范

2. **可复用性检查规范** - 精细、完善
   - 如何识别重复代码
   - 如何提取可复用逻辑
   - 具体的代码示例

3. **硬编码检查规范** - 精细、完善
   - 哪些值不能硬编码
   - 如何使用配置文件
   - 具体的代码示例

4. **函数设计规范** - 精细、完善
   - 单一职责原则
   - 参数设计
   - 返回值设计
   - 具体的代码示例

5. **组件设计规范** - 精细、完善
   - Props 设计
   - Emits 设计
   - 职责划分
   - 具体的代码示例

6. **代码规范检查** - 根据项目的 .eslintrc 调整
   - 分析当前的 ESLint 配置
   - 补充缺失的规则
   - 生成调整建议

### 第三步：测试和验证

用新项目测试：
1. 让 Cline 生成一个新功能
2. 对照规范检查代码质量
3. 记录问题和改进建议
4. 迭代优化规范
