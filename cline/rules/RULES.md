# Cline 业务开发规则

## 核心原则

1. **低耦合** - 模块之间保持松散耦合，通过接口通信，避免直接依赖
2. **避免魔法数字** - 所有业务常量必须定义在常量文件中，禁止硬编码
3. **业务逻辑严谨** - 类和方法设计要符合单一职责，接口设计要清晰
4. **代码简洁** - 避免过度工程，不必要的抽象不要做
5. **易于维护** - 代码要容易被理解和修改，后续迭代不容易出错

---

## 具体规则

### 1. 低耦合规则

- 模块之间通过接口通信，不直接引用实现类
- 共享数据通过事件或状态管理器，不直接修改对方状态
- 独立功能封装为独立模块，不与其他模块混在一起
- 组件之间通过 props/emit 通信，不直接修改对方数据

### 2. 避免魔法数字

- 所有业务常量必须定义在 constants/ 目录
- 状态值、类型值必须使用枚举或常量
- API 地址、超时时间等必须从配置文件读取
- 禁止在代码中出现裸值：1, 2, 3, 'pending', 30000

### 3. 业务逻辑设计

- 类名和方法名要能表达业务含义
- 方法只做一件事，方法长度不超过 50 行
- 复杂业务逻辑提取为独立的服务类或工具函数
- 接口设计要考虑扩展性，避免过度设计

### 4. 代码简洁

- 能用简单方式实现就不要复杂化
- 不必要的注释不要写
- 不必要的抽象不要做（DRY 但不要过度抽象）
- 保持代码线性，不要嵌套过深

### 5. 易于维护

- 代码逻辑要清晰，容易理解
- 变量和方法命名要清晰
- 复杂逻辑要有注释说明为什么这么做
- 考虑后续可能的扩展，但不要提前优化

---

## 你要避免的问题

- ❌ 直接在组件里写复杂业务逻辑
- ❌ 两个不相关的功能写在同一个方法里
- ❌ 硬编码的状态值：if (status === 1)
- ❌ 过度抽象：为了"可扩展"设计一堆接口和基类
- ❌ 方法过长：一个方法做了太多事情
- ❌ 嵌套过深：if 里面 if 里面 if
