# Cline 业务开发规则

版本：v0.1（草案）

目的
----
建立可执行、易检验的业务开发规范：降低耦合、避免硬编码、提升可维护性与审查效率。

核心原则（必读）
----------------
- 低耦合 — MUST：模块间通过明确接口/服务/事件交互，禁止直接依赖实现细节。  
- 单一职责 — MUST：类/方法应只做一件事，便于理解、测试与复用。  
- 避免魔法数字 — MUST：所有业务常量集中管理（constants/），使用枚举或常量对象。  
- 可读与简洁 — SHOULD：优先简单可读实现，避免不必要的复杂抽象。  
- 可维护性 — SHOULD：命名清晰、边界情况明确，复杂逻辑说明“为什么这样做”。

具体规则（格式：规则 — 等级 — 说明 — 检查点）
--------------------------------------------
1. 低耦合 — MUST  
   说明：模块之间仅通过接口/服务/事件通信，内部实现隐藏。  
   检查点：是否存在跨模块直接访问实现类或内部状态的引用？

2. 业务常量集中管理 — MUST  
   说明：状态值、类型、超时、API 地址等禁止裸写在业务代码中。  
   检查点：常量是否位于 constants/ 或配置文件；是否使用枚举。

3. 单一职责与方法长度限制 — MUST / SHOULD  
   说明：方法应表达单一动作，建议不超过 50 行、参数不超过 5 个；复杂逻辑拆分为服务/工具。  
   检查点：是否存在超长函数或多职责函数。

4. 命名规范 — MUST  
   说明：变量、方法、类应清晰表达业务意图（如 fetchUser、submitOrder）。  
   检查点：是否存在无意义命名（arr1、tmp）或混淆命名。

5. 控制副作用 — MUST  
   说明：副作用应显式、可控；UI 组件尽量避免直接包含大量业务逻辑。  
   检查点：组件内是否包含未经封装的业务实现。

6. 抽象与复用策略 — SHOULD  
   说明：以“3 次重复”为提取复用参考阈值；避免为假想需求过早抽象。  
   检查点：是否为少量重复创建复杂抽象。

7. 错误处理与可见性 — MUST  
   说明：异步操作需显式捕获并向用户/监控暴露可理解错误信息。  
   检查点：异步调用是否有 try/catch 或统一处理；是否有用户提示。

8. 代码审查与例外流程 — SHOULD  
   说明：规则分级（MUST/SHOULD/OPTIONAL），例外需在 PR 中说明并指定审批人。  
   检查点：PR 是否包含例外说明与审批记录（如适用）。

可执行要求（快速落地）
---------------------
- 每条 MUST 尽量对应可自动校验项（ESLint/TypeScript 规则或 CI 脚本）。  
- 文档需标注规则等级以便自动化处理。  
- 为每条规则准备简短检查清单，供 Code Review 使用。

维护与变更
---------
- 每次修改须更新版本号与 CHANGELOG（记录变更理由与影响范围）。  
- 规则变更由架构/维护团队审批，重大变更需在 PR 中评估影响并通知相关团队。
